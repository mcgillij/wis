Executive Summary

Use constellations as regional navigation-scale hero landmarks, not as literal nearby meshes. A constellation should be a deterministic data object that binds several real star systems, a 2D/3D asterism pattern, navigation metadata, and billboard/impostor visual presets. From far away the player sees one stylized billboard hero. As distance and approach increase, the shader reveals named anchors, lanes, and brighter member stars. When the player is inside one member system, that system's real primary star is one member of the constellation; all other members are rendered as fake far star impostors placed on the local sky pattern.

  • Best model: region-level ConstellationDefinition plus per-system runtime projection state.
  • Best rendering path: extend the existing billboard hero manifest architecture from docs/done/billboard_hero_tasks.md; do not invent a separate renderer.
  • Closest-range rule: exactly one real member star is the current system's actual primary/binary component; every other visible point is an impostor using the regular/exotic far-impostor pipeline.
  • Player promise: every highlighted point that receives a label/target affordance must correspond to a reachable system or a clearly marked lore-only mirage.

Design Model

The constellation should be generated at a larger scale than an individual system, then projected into each member system. Store enough information for deterministic regeneration from galaxy seed and coordinates, while avoiding cross-system runtime synchronization.

Visual Model by Range

  • Galaxy / map scale: thin constellation glyph, named region badge, optional route chain. Use low-frequency icons; avoid star-field noise.
  • Far system sky: one billboard hero using the existing billboard_hero_constellation.gdshader. Show broad lane/trail, faint pattern points, no exact targeting except scan hint.
  • Mid range / repeated jumps toward it: increase reveal, apparent_scale, stage, and edge visibility. Anchor stars separate from decorative background speckles. A scan can name 1-3 reachable member points.
  • Near member approach: brightest target point blooms in the destination star color. Lines become route hints rather than physical filaments.
  • Same-system closest range: the local star scene is real. Other member stars use far impostor nodes/billboards pinned to the projected pattern with spectral color, bloom, flare shape, and a tiny parallax hint, but no full physical duplicate bodies.

Closest-Range Behavior

Resolve membership before rendering: if the active system ID equals member_system_ids[i], set current_member_index = i and force the real star recipe to match member_star_types[i] within allowed stellar-generation constraints. For binaries, allow the constellation member to bind to the primary unless the definition explicitly marks a secondary. All other members become impostors:

  • Use RegularStarFarImpostor for O/B/A/F/G/K/M and giant variants, and ExoticStarFarImpostor for neutron/black-hole/dark-star-style members.
  • Place impostors by projecting pattern_points onto a local sky plane in the constellation facing direction, not by spawning far physical systems.
  • Scale brightness from intended apparent magnitude, current member relationship, and accessibility state. Reachable targets get a stable scan label; decorative support stars stay unlabeled.
  • Do not allow impostors to cast local shadows, affect gravity, or create collision/hazard volumes unless the player jumps to that member system.

Procedural Generation Rules

  1. Region selection: sample rare constellation anchors from galactic coordinates; bias to faction borders, old clusters, pilgrimage paths, nebula edges, or hidden-system chains.
  2. Member count: 3-7 reachable members for gameplay readability; add optional decorative minor points only in shader.
  3. Pattern grammar: choose from line, hook, crown, gate, spiral, broken triangle, river/stream, and shattered cluster templates. Then jitter points deterministically.
  4. Star type budget: require at least one visually distinct color contrast, e.g. K/G anchor plus B/A bright point, or M dwarf plus white dwarf remnant. Keep exotic members very rare.
  5. Rarity: most systems should have none; a region may contain one landmark constellation and a few local decorative clusters.
  6. Stability: derive all IDs, member indices, and visual seed variants from galaxy seed + anchor coords + sorted member coords. Save only discovery/label state.

Shader / Material Recommendations

  • Keep billboard_hero_constellation.gdshader as the first family. It already exposes star_density, lane_strength, trail_strength, twinkle_strength, reveal, stage, and approach.
  • Add pattern support without large uniform arrays: encode up to 8 member points and edges in a small 1D lookup texture or packed vec4 pattern_points_0..3 uniforms for V1.
  • Separate decorative star noise from member anchor stars. Member anchors need stable positions, sizes, labels, and target hit proxies.
  • Add uniforms: member_count, selected_member_index, edge_visibility, label_pulse, route_hint_strength, same_system_member_index.
  • For accessibility, support a high-contrast mode where edges and anchor stars are hue-separated from the background sky.

Navigation UX Recommendations

  • Scanning the hero reveals: constellation name, nearest member, current-member status, and 1-2 reachable points.
  • Focus cycling should snap between anchor points, not decorative stars. The selected anchor pulses and displays distance/jump count.
  • Map and backdrop must use the same pattern orientation; if the sky projection is stylized, label it as a navigational projection.
  • Use verbs carefully: “Aligned with The Harp / Blue Anchor reachable” is safer than “the star is physically ahead 4 ly” unless the route solver confirms it.

Risks and Pitfalls

  • Parallax confusion: fake same-system impostors must not look like nearby physical suns.
  • Unreachable decoration: never give target affordances to non-member shader speckles.
  • Clutter: cap hero landmarks per sky direction; constellation edges should fade behind brighter local phenomena.
  • Save determinism: persist discovered labels, not generated pattern geometry.
  • Over-promising astronomy: real constellations are line-of-sight projections; game constellations are deliberate navigable asterisms.

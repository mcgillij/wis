# 3d-space: Server-Side Simulation Hooks Research how the `/home/j/3d-space` simulation layer should hook into main game systems from the server side. ## Goal Produce a standalone HTML report proposing clean server-authoritative integration points between simulation state and live gameplay systems. ## Focus questions - Which main game systems should consume simulation data: trade markets, NPC spawning, ambient traffic, pirate/security encounters, faction reputation, station services, mission generation, map UI, new…
Research basis
Local repository inspection of /home/j/3d-space; no external web research was used.

Executive Summary

The server-side hook surface is already partially in place. Dedicated server startup configures ServerRuntimeServices with server_simulation_persistence; TradeService can query and record simulated market state; FeedEventService can project unconsumed simulation events; NpcCombatManager centralizes authoritative NPC combat/damage. The missing piece is a small, explicit set of hook contracts for live gameplay events to write facts without coupling to database internals.

The safest design is to use server-authoritative hook points at event completion, damage/destruction, market transaction, scan completion, and route delivery. These should enqueue/record durable facts, not directly mutate strategic state.

Key Findings

  • ServerRuntimeServices has set/get_simulation_persistence and configure_for_dedicated_server instantiates server_simulation_persistence.
  • server_simulation_persistence writes player_contribution_facts, upserts station_market_state, adjusts station inventory, records market_trade_facts, reads station market history, and exposes conflict feed data.
  • server_database schema includes simulation_jobs/ticks, station_market_state/history, trade_route_state, market_trade_facts, conflict_zones, simulation_events, and player_contribution_facts.
  • TradeService uses /root/ServerRuntimeServices.get_simulation_persistence() to fetch snapshots and history and to record market changes.
  • DedicatedServer calls FeedEventService.project_unconsumed_simulation_events(), meaning simulation_events can already become user-facing feed material.
  • NpcCombatManager.apply_damage routes authoritative NPC damage through runtime controllers, making it a good hook point for npc_damaged/npc_destroyed facts.

Detailed Notes

Recommended hook contract: SimulationFactWriter.record_fact(kind: String, scope: Dictionary, subject: Dictionary, value: float, payload: Dictionary) -> Dictionary. The writer should resolve player ID, system ID, faction IDs, route IDs, actor IDs, timestamps, and source event IDs. It should choose the correct persistence method or fallback queue.

Hook points: 1) TradeService.execute_buy/execute_sell/apply_npc_delivery record market_trade facts with station, route, commodity, quantity, price, and actor/player source. 2) NpcPiracyEventDriver stage transitions record piracy_demand, piracy_compliance, cargo_jettisoned, cargo_collected, piracy_refused, and raider_destroyed. 3) NpcCombatManager.handle_npc_destroyed records npc_destroyed with actor/faction/role/attacker/source event. 4) BountyHuntMissionDriver completion records bounty_completed and reputation payout. 5) FactionConflictEventDriver records conflict_kill and conflict_resolution contributions. 6) DiscoveryManager.stage_star_science_facet records scan_revelation only if a later strategic layer needs exploration pressure.

Avoid hook side effects that directly update markets/faction control from live code. Live code should record facts and display immediate local outcome. The strategic tick should compute durable economic/faction changes. This prevents mission driver code from becoming a second economy simulator.

Network authority: Hooks should execute only on the authoritative peer/server. Existing code frequently checks authoritative event mutation and uses server services; keep that pattern. For local/offline play, the same writer can point at local persistence.

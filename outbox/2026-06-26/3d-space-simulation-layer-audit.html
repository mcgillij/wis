# 3d-space: Simulation Layer Audit Audit the simulation layer in `/home/j/3d-space` and explain what it currently does, what state it owns, and where it is underused. ## Goal Produce a standalone HTML report that maps the current simulation layer and identifies the highest-value ways to put it to better use. ## Focus questions - What scripts/classes/services make up the simulation layer? - What domain state does it already track: NPC routes, trade deliveries, factions, stations, resources, encounters, economy, popu…
Research basis
Local repository inspection of /home/j/3d-space; no external web research was used.

Executive Summary

The project already has the bones of a rich NPC/simulation layer: active-system NPC orchestration, lightweight-versus-combat runtime promotion, trade facts, conflict facts, public simulation events, and feed projection all exist. The current gap is less “missing architecture” than “missing closed-loop pressure”: many systems can record facts or drive one-off events, but fewer systems visibly feed those outcomes back into persistent faction control, route risk, station scarcity, patrol density, or player-facing mission offers on a predictable cadence.

The strongest near-term strategy is to standardize the contract between live gameplay events and the strategic tick. Treat piracy, bounty kills, conflict-zone participation, NPC deliveries, and player trading as typed facts; let the nightly/strategic worker consume those facts; then project concise results back to markets, map overlays, feed cards, and job offers.

Key Findings

  • The server database schema includes simulation_jobs, simulation_ticks, system_control_state, station_market_state, trade_route_state, market_trade_facts, station_market_history, conflict_zones, simulation_events, and player_contribution_facts, so persistence is already designed for strategic ticks and feed/event outputs.
  • ServerRuntimeServices configures server_simulation_persistence on dedicated servers, and TradeService queries that persistence for market snapshots/history and records market trade facts.
  • NpcSimulationService and NpcWorldOrchestrator create a clear boundary for active-system NPC contacts, runtime-mode requests, hails, and combat promotion; this is the right façade to avoid game systems reaching into simulation internals.
  • NpcCombatManager is already a canonical damage/combat registry, which gives combat events a reliable place to emit facts rather than duplicating kill/assist logic in every mission driver.
  • FeedEventService / server_feed_projection_backend can project public simulation_events into story candidates and feed articles, which is a strong user-facing outlet for strategic consequences.
  • The audit found multiple rich point systems, but not a single obvious “ledger of outcomes by cause” visible to the player: route disruptions, demand spikes, faction control changes, bounty pressure, and NPC losses need consistent UI surfacing.

Detailed Notes

Keep NpcSimulationService as the live-world façade and ServerRuntimeServices/server_simulation_persistence as the durable fact backend. Do not let every event driver write bespoke tables. Instead, define a small set of simulation fact types: market_trade, cargo_loss, piracy_compliance, piracy_refusal, npc_destroyed, bounty_completed, conflict_kill, conflict_resolution, scan_revelation, and route_delivery.

The database schema already separates raw facts from projected events. Lean into that split. Raw facts should be append-only, low-level, and hard to lose; simulation_events should be player-facing or system-facing conclusions generated by a tick; feed/event cards should be downstream projections from simulation_events.

NpcPiracyEventDriver is especially valuable because it touches cargo, dialogue, AI, combat escalation, world drops, and completion. It should become a reference implementation for how a live event records both “what happened” and “why it matters.” For example: compliance should reduce immediate danger but increase local pirate confidence; refusal and pirate destruction should reduce pirate pressure and raise bounty/conflict hooks; cargo recovered by pirates should become a supply-side market fact in pirate stations or black markets.

The star scanner already creates high-quality, per-mode discovery facets. That discovery pipeline can become a model for simulation facts: deterministic payloads, quality multipliers, staged player state, sale/publish hooks, and feed projection.

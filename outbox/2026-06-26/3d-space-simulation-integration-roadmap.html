Research basis
Local repository inspection of /home/j/3d-space; no external web research was used.

Executive Summary

The integration path should convert the existing systems into a closed loop: live gameplay emits facts, a strategic tick consumes facts, simulation state changes, and feed/jobs/UI expose those changes back to the player. The repository already has nearly every component needed for this architecture.

The highest leverage work is standardization: one fact-writing path, one strategic tick consumption contract, one projection path to feed/jobs, and one debug surface for verification.

Key Findings

  • NpcWorldOrchestrator registers/deregisters active systems with NpcSimulationService and forwards runtime-mode/hailed signals, which is a clear live-world bridge.
  • NpcCombatManager registers NPC runtimes/groups, changes group runtime states, applies authoritative damage, and provides combat snapshots; use it for combat fact hooks.
  • TradeService and server_simulation_persistence already join player/NPC market activity to station_market_state and market_trade_facts.
  • Feed projection can consume public simulation_events and create story candidates, which is the correct downstream narrative channel.
  • ActiveJobsHudController and JobBoardPresenter can merge active system events, bounty job entries, station offers, and conflict/pirate raid offers into actionable UI.

Detailed Notes

Step 1 — Define simulation fact schema in code. Even with flexible payload_json, codify required keys for each fact type in one place: event_id, system_id, world_instance_id, route_id, actor_id, player_id, faction_id, commodity_id, quantity, value, outcome, occurred_at. Add validation warnings in debug builds.

Step 2 — Add fact hooks to four vertical systems: TradeService for market_trade, NpcPiracyEventDriver for piracy/cargo/raider outcomes, BountyHuntMissionDriver for bounty completion, and FactionConflictEventDriver for conflict contribution/resolution. Use authoritative guards.

Step 3 — Implement strategic tick consumption. The tick reads unconsumed facts, updates station_market_state, trade_route_state, conflict_zones, and system_control_state, writes simulation_events, then marks consumed_tick_id. Keep tick deterministic per date/tick_id where possible.

Step 4 — Project simulation_events. FeedEventService already handles unconsumed public events. Extend projections for route disruption, market shortage/surplus, pirate activity, bounty pressure, and faction control changes.

Step 5 — Surface in gameplay. Add route-risk badges, station shortage/surplus explanations, job-board offers based on recent events, and local map overlays for active conflict/piracy pressure.

Step 6 — Verify with scenario tests. Create integration tests that seed facts, run a tick, assert state changes, project feed entries, then assert job-board/active-jobs entries reflect them.

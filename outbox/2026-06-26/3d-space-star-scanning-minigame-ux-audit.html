Research basis
Local repository inspection of /home/j/3d-space; no external web research was used.

Executive Summary

The star scanner is feature-rich but risks overwhelming players because the scientific controls, meters, lock threshold, alignment validity, mode switching, and external HUD suppression all happen at once. The UX should emphasize a small repeated pattern: pick a mode, tune three controls, watch meters, hold lock, receive a readable finding.

The implementation has good hooks for improvement: quality details, mode meters, directions text, scan_mode_changed, scanner_opened/closed, and DiscoveryManager staging. The audit recommends better hierarchy and result communication rather than wholesale rewrites.

Key Findings

  • The scanner UI is built dynamically in StarScannerOverlay._build_ui with a mode label, mode buttons, status label, directions toggle, meter rows, control rows, and readout label.
  • Status states distinguish no valid target, alignment lost, scan archived, and tune sustained lock.
  • Meter rows show Signal, Noise, Lock, Progress, plus mode-specific meters from quality details.
  • Control rows show mode-specific labels: Wavelength/Focus/Gain, Frequency/Bandwidth/Squelch, Baseline/Phase/Delay, Exposure/Gate/Shield Bias.
  • The readout currently summarizes LOCKED/ACTIVE, control names, and Quality %. This is useful but not enough to explain next action.
  • HudOverlay suppresses regular HUD chrome while scanner is active and enables ScannerVisionOverlay, which helps focus but also removes ambient context.

Detailed Notes

UX issue 1 — Objective clarity. The current status text “tune sustained lock” is concise but not instructive for first use. Add a persistent objective line: “Raise Lock above 82%, then hold alignment until Progress fills.” This should remain visible even when directions are collapsed.

UX issue 2 — Meter meaning. Signal/Noise/Lock/Progress are readable, but mode-specific meters need interpretation. Add a small hint under the worst meter: “Noise Floor high: raise Squelch” or “Phase Error high: adjust Phase/Delay.” The evaluators already return mode_meters; map meter names to likely controls.

UX issue 3 — Directions panel size. The panel increases the scanner panel offset from 238 to 350. That is useful, but the expanded panel should not hide essential meters on smaller screens. Consider a side panel or compact tooltip layout.

UX issue 4 — Completion feedback. scan_completed is emitted and DiscoveryManager stages data, but the scanner should show a result summary card: mode, quality, fields revealed, sale/value effect, and next recommended mode.

UX issue 5 — Archived scans. “scan archived” tells the player a mode is done, but not whether all modes are done or what remains. Add mode buttons with completed checkmarks and a “2/4 analyses complete” header.

UX issue 6 — Alignment loss. Alignment validity is updated from HudOverlay by comparing current scanner context to reticle/target. When alignment is lost, show a reticle arrow or “re-center target” prompt rather than only pausing/decaying progress.

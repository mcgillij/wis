Research basis
Local repository inspection of /home/j/3d-space; no external web research was used.

Executive Summary

The star scanner already has a solid technical base: four modes, three controls per mode, lock-quality functions, sustained-lock completion, mode switching, pointer/axis/button input, a directions panel, and post-process mode visuals. The controls roadmap should make the minigame easier to learn while preserving mastery and mode identity.

The main recommendation is to add progressive disclosure: show a simple objective first, reveal mode-specific hints on demand, provide subtle feedback about which control is wrong, and make high-quality lock feel earned rather than random.

Key Findings

  • StarScannerOverlay defines LOCK_THRESHOLD=0.82, LOCK_SECONDS_REQUIRED=1.35, PROGRESS_DECAY=0.48, control step constants, and separate input actions for primary/secondary/tertiary adjustments.
  • The four modes are optical spectroscopy, radio spectroscopy, radio interferometry, and X-ray spectroscopy, each with controls/meters/directions and a distinct tint.
  • The UI has a Directions button/panel that displays mode-specific bullet instructions plus generic controls text.
  • Mode actions already support shortcuts: optical adjusts gain, radio nudges toward signal center or raises squelch, interferometry cycles baseline/adjusts delay, X-ray targets emission window or shield bias.
  • The scanner emits scan_mode_changed with quality, which drives ScannerVisionOverlay quality/tint changes through HudOverlay.
  • Completion emits entity_id, display_name, scan_mode, scan_quality, and star_data, which DiscoveryManager turns into star-science facets.

Detailed Notes

Control Phase 1 — Learnability. On first open, show a compact “Goal: tune Lock above 82% and hold for 1.35s.” Then show the three active controls with arrows or +/- buttons beside each. Keep the Directions panel, but make it mode-contextual and shorter by default. Add an optional “advanced details” expander with the longer scientific hint text.

Control Phase 2 — Feedback. When lock is below threshold, indicate the dominant limiter: “Line Fit low,” “Noise high,” “Phase error,” or “Dose unsafe.” The quality evaluators already return mode_meters, so this can be computed without exposing formulas. Highlight the control row most likely tied to the lowest meter.

Control Phase 3 — Input ergonomics. Keep number keys/mode next/previous but add consistent labels: primary action, secondary action, analog stick/pointer drag, and keyboard increments. For controllers, show the bound glyph/display name from InputActions if available. Ensure cancel closes scanner but does not silently discard completed modes.

Control Phase 4 — Mastery. Add optional “fine tune” modifier that reduces CONTROL_STEP for precision, and a “coarse tune” modifier for fast movement. Do not make lock solutions fully automatic; mode action nudges can teach direction but should not solve all controls instantly.

Control Phase 5 — Reward clarity. Before scanning, show what mode reveals. After scanning, show what was gained and how quality affected precision/value.
